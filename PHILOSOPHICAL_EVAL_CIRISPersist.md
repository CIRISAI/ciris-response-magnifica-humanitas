## Gap E-2: Affected-Party-Input Ledger

**Version**: 1.0
**Date**: 2026-05-25
**Evaluator scope**: CIRISPersist MISSION only. Closed-set verdict.

---

### Verdict: SUBSTANTIAL_EDIT

---

### Analysis

**Is affected-party input categorically the same kind of "signed evidence" CIRISPersist already stores?**

No. CIRISPersist stores *machine-originated signed evidence*: reasoning traces emitted by agents, audit log entries generated by the system, memory graph state, telemetry. Every record is authenticated by a federation key — a CIRISVerify-rooted identity that the corpus can verify cryptographically. The ingest pipeline is: `bytes → schema parse → verify → scrub → decompose → insert`. Unverified bytes never reach the backend; the verify gate is unconditional.

Affected-party input is *testimonial input from non-federation participants* — humans outside the trust graph who have no CIRISVerify key, no Ed25519 signing capability, no dedup key, no schema version. They cannot pass `verify_strict`. Storing their input requires either (a) a new trust model for unsigned human testimony, or (b) a wrapper that signs it on their behalf (introducing an attestation-by-proxy problem CIRISPersist's MISSION explicitly avoids — it never confers trust, only records what the chain recorded). This is a categorically different evidence class, not an extension of the existing one.

**Does adding it conflict with CIRISPersist's substrate-only philosophy?**

Yes, structurally. CIRISPersist §1.3 and §1.4 define the apophatic bound sharply: persist does not reason, does not score, does not verify identity — "it carries state for the peers that do." Accepting and storing human testimonial input requires either:

- Interpreting what counts as valid affected-party testimony (reasoning), or
- Making a trust determination about unkeyed human actors (trust oracle), or
- Wrapping unsigned content with a surrogate signature (rolling its own trust model).

All three violate stated apophatic bounds. The MISSION's philosophy is substrate-only: CIRISVerify makes things *evidence*; CIRISPersist makes evidence *last*. Testimonial input from non-federation parties is not evidence in CIRISPersist's sense until something upstream makes it so — and nothing in the current federation does.

**Is there a philosophical gap in the MISSION?**

Yes. CIRISPersist MISSION §1.1 grounds everything in M-1 (sustainable adaptive coherence) and names Justice as a load-bearing principle — specifically that durable tamper-evident storage on both Postgres and SQLite ensures no peer is second-class. But the Justice principle the encyclical demands in §§216–217 (affected parties must have direct input, not merely post-hoc transparency) requires the corpus to be *receivable by* the persons whose lives it concerns. CIRISPersist's current philosophy makes the corpus legible *about* those persons; it has no concept of evidence *from* those persons. The MISSION does not acknowledge this asymmetry, does not name it as a boundary, and does not bow out (which would be the honest move per MISSION.md §1.3's grounding posture). It simply has no position. That absence is load-bearing because it would need to be resolved — either CIRISPersist extends its trust model, or a new federation component handles it, or the Accord names the gap as a structural limit — before E-2 can be implemented honestly.

**Rationale** (≤30 words): Testimonial input from unkeyed humans cannot pass `verify_strict`; storing it requires a new trust model the MISSION neither provides nor disclaims as out of scope.

---

### Closed-set verdict

**SUBSTANTIAL_EDIT**

A clarifying sentence is insufficient. The MISSION requires a new section or apophatic extension naming testimonial-from-non-federation-parties as either (a) out of scope with a handoff to CIRISNode, or (b) a new trust surface requiring explicit design. Without it, E-2's implementation at CIRISPersist is either a mission violation (trust-oracle creep) or an undocumented deviation from the substrate-only philosophy.

### Recommended edit target

`MISSION_CIRISPersist.md` §1.4 (Apophatic bound): add a fifth refusal — "Not a testimonial receiver. CIRISPersist stores evidence that the federation's trust chain has authenticated. Human testimony from non-federation participants is not CIRISPersist-receivable evidence; routing affected-party input into the durable corpus requires an upstream attestation decision by CIRISVerify or CIRISNode, not by persist." If E-2 is implemented at CIRISNode (the interim assignment in GAPS.md), this refusal makes the interim correct by design, not by default.
